# Provenance Note For The Current Public Benchmark Bundle

Date: 2026-03-08

## Why the commit hashes do not all equal the current tree HEAD

This public repository currently stores:

1. the benchmark harness code, and
2. checked-in benchmark outputs generated by that harness.

Each raw run JSON records the Git commit visible **at execution time** in:

- `commit_hash`
- `backend_commit`

If new outputs are generated and then committed afterward, the recorded run hash naturally points to the immediately preceding harness commit rather than to the final commit that stores the output files.

That is the workflow state of the current bundle.

## Current provenance groups

At the time of writing, the tracked outputs fall into at least two commit groups:

- earlier shared-input parity reruns generated under `2763823...`
- the newer post-jump repeated rerun at `scale = 4520`, `k_run = 18`, `repeats = 3`, generated under `2f9b4bb...`

The tree that stores those files may therefore have a later commit hash than the raw run JSON embedded inside the files.

## What this does and does not mean

This does **not** mean the outputs are anonymous or detached from the repository history.

It does mean that a reviewer should interpret the raw run JSON as:

- produced by the immediately preceding audited harness commit,
- then committed into a later tree for publication.

So the current bundle is still reviewable, but its provenance is grouped by generation commit rather than by a single tree-wide HEAD match.

## Conservative interpretation rule

For review purposes, treat the raw run JSON as authoritative for:

- which harness commit produced the measurement,
- rust version,
- machine profile,
- and measured metrics.

Treat the README / tree HEAD as authoritative for:

- which files are currently published together,
- which result families are being presented for review.

## Planned clean-up direction

A stricter future publication workflow could avoid this mismatch by:

1. pinning review to the exact generation commit for each output family, or
2. publishing outputs under a release/tag workflow that freezes both harness and results together.

That stricter workflow has not yet been adopted in this repository.

## Bottom line

There is a real provenance mismatch at the tree-HEAD level, and reviewers are right to notice it.

The current package should therefore be read as:

- a reviewable public benchmark bundle with explicit per-run generation commits,
- but not yet a perfectly single-snapshot artifact in which every tracked output was produced at the same final tree HEAD.
